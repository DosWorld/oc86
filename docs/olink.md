# OLINK — RDOFF2 Smart Linker for MS-DOS

Source: `src/olink.c` + `src/olink.h`  
Binary: `olink` (Linux/Mac) / `olink.exe` (DOS)  
Build: `cd src && make` (gcc) or `wmake -f Makefile.wat` (Open Watcom)

---

## Usage

```
olink [options] <input> ... <output.exe>

  -L path     Add library search path (repeatable)
  -h, -?      Show help
```

The last non-option argument is always the output filename.

Typical workflow:
```bash
oc -entry Run MyMod.Mod                     # compile → MyMod.om
olink MyMod.om -L lib SYSTEM.om myapp.exe  # link → myapp.exe
```

---

## Input formats

| Argument   | Meaning |
|------------|---------|
| `file.om` | ustar tar archive produced by `oc` — only `.rdf` members are linked; `.def` members are skipped |
| `@listfile` | text file listing inputs one per line (whitespace trimmed, blank lines ignored) |

Standalone `.rdf` files are **rejected** with a fatal error. All RDOFF2 objects must be packed into `.om` archives by `oc` before linking. This eliminates linking-order bugs caused by mixing standalone objects with archives.

Format detection: `.om` files are identified by the `ustar` magic at offset 257, or by `.OM` extension (case-insensitive). A bare `RDOFF2` signature at offset 0 triggers the rejection error.

---

## Library search order

When an input name is not found as-is, the linker searches in this order:

1. Current directory (or absolute path as given)
2. Paths added with `-L` (in command-line order)
3. `OBERON_LIB` environment variable — colon-separated on Unix, semicolon-separated on DOS

Arguments starting with `/` on Unix are treated as file paths, not DOS-style option flags. Only `-` prefix triggers option parsing.

---

## Smart linking

Only modules reachable from the symbol `start` are included in the output.

- `start` is exported by `MAIN.RDF`, generated by `oc -entry ProcName`.
- Reachability is computed by `olink_smart_link`: marks the module owning `start`, then recursively marks every module whose GLOBAL is imported by an already-marked module.
- Modules not reachable from `start` are scanned (their GLOBALs are registered) but their code/data is not emitted.
- Unresolved IMPORT in a reachable module is a fatal linker error.

---

## Memory model

olink produces DOS executables with the following segment layout:

- **One code segment per .om group** — all `.rdf` members from the same `.om` file are packed into one paragraph-aligned code segment (2-byte word alignment between members). Each group has one CS. A FAR CALL `9A oo oo ss ss` between groups uses a SEGRELOC on `ss ss` to set the target group's CS; the offset `oo oo` is group-relative (= `sym->offset + module_code_ofs_within_group`). Standalone `.rdf` files each form their own single-module group.
- **One combined data segment** — all module data and BSS are merged. DS is the same for all modules and never changes at runtime.
- **Separate stack segment** — SS points to a distinct region after data+BSS. SS ≠ DS at runtime.

```
At runtime (all values are absolute segments; MZ loader adds load_seg):

  CS = load_seg + hdr_para + entry_cs_para   ; entry group's code segment
  DS = load_seg + hdr_para + code_para       ; combined data segment
  SS = load_seg + hdr_para + code_para + data_bss_para  ; stack segment

  IP = entry_ip   (group-relative offset of "start")
  SP = STACK_SIZE - 2
```

SYSTEM_INIT (called at program entry, generated by `oc -entry`) sets DS from the SEGRELOC-patched data segment paragraph in the code; it cannot derive DS from SS.

## Output: DOS MZ executable

### Memory layout at runtime

```
load_seg + hdr_para + 0               code group 0  (paragraph-aligned start)
                                         module A (2-byte aligned within group)
                                         module B (2-byte aligned, follows A)
                                         ...
load_seg + hdr_para + group1_para     code group 1  (paragraph-aligned start)
                                         ...
  ...
load_seg + hdr_para + code_para       data segment:
                                        [0 .. total_data_len-1]  initialised data
                                        [total_data_len ..]       BSS (zero-filled)
load_seg + hdr_para + code_para
                    + data_bss_para   stack segment (STACK_SIZE bytes, grows down)

SP = STACK_SIZE - 2
IP = entry_ip   (group-relative offset of "start" = sym->offset + module_code_ofs_within_group)
CS = hdr_para + entry_cs_para  (group_code_para of entry group; relative to load_seg)
SS = hdr_para + code_para + data_bss_para  (relative to load_seg)
DS = not set by MZ loader; SYSTEM_INIT loads it from the patched DS paragraph
```

### MZ header fields

| Field | Value | Notes |
|-------|-------|-------|
| signature | `MZ` (0x5A4D) | |
| pages / partial page | computed | from total exe size |
| reloc count | mz_reloc_cnt | one entry per SEGRELOC |
| header size | hdr_para paragraphs | 28-byte fixed header + reloc table, rounded up |
| min_alloc | 0 | no extra allocation; image size is exact |
| max_alloc | 0xFFFF | give program all available memory |
| SS | code_para + data_bss_para | stack segment after data+BSS, relative to load image start |
| SP | STACK_SIZE - 2 | |
| IP | entry_ip | group-relative offset of `start` (= sym->offset + module_code_ofs_within_group) |
| CS | entry_cs_para | `group_code_para` of the entry group, relative to load image start |
| reloc table offset | MZ_HDR_BYTES (28) | immediately after fixed header |

### Segment layout in the file

```
[hdr_para * 16 bytes]   MZ header + reloc table + padding
[total_code_len bytes]  code groups (paragraph-aligned between groups; 2-byte aligned within)
[total_data_len bytes]  combined data segment (initialised data only)
```

BSS and stack are not written to the file — DOS zero-extends the image automatically.

## Linking pipeline

The linker runs these phases in order:

### 1. Scan (`scan_rdf` / `scan_tar`)

For every input:
- Open the file; detect RDOFF2 or tar format.
- For tar: iterate 512-byte ustar headers; register each `.rdf` member with its byte offset and size.
- For each RDOFF2 module: read the header records to collect GLOBAL symbols and BSS size; read segment blocks to record code/data sizes.
- All GLOBALs are added to the global symbol table (`ls->sym_head`).

### 2. Smart link (`olink_smart_link`)

- Locate the `start` symbol; fatal error if absent.
- Depth-first mark: open the RDOFF2 header of each reachable module, walk IMPORT records, look up each imported name in the symbol table, mark the owning module, recurse.
- Unresolved IMPORT in a reachable module → fatal error.

### 3. Layout (`olink_calculate_layout`)

- Walk used modules in scan order, grouping by `group_id` (all members of one `.om` have the same `group_id`; standalone `.rdf` each get their own).
- For each group: paragraph-align the code base; this becomes `group_code_para` for all members. Within the group, pack modules with **2-byte word alignment** and assign `final_code_ofs` for each. After the group, `code_base` is left at the next byte after the last member (next group paragraph-aligns at its start).
- Data and BSS offsets are 16-byte aligned per module across all groups (unchanged).
- Set `group_code_para = group_code_start >> 4` for every module in the group.
- Compute `total_code_len` (paragraph-rounded), `total_data_len`, `total_bss_len`.
- Fatal error if data+BSS exceeds 65520 bytes (must fit in one segment).

### 4. Link (`olink_perform_linking`)

For each used module, three sub-passes:

**Load**: copy module's raw code bytes into `code_buf[final_code_ofs..]` and data bytes into `data_buf[final_data_ofs..]`.

**Pass 1 — collect IMPORTs**: walk header IMPORT records; build a per-module `id → TSymbol*` map (`TImportNode` list). Required because RELOC records may reference import IDs that appear later in the header.

**Pass 2 — apply RELOCs**: walk header RELOC and SEGRELOC records:

- **RELOC** (type 1): patch a 1- or 2-byte field. The `rseg` byte low 6 bits = physical segment (0=code, 1=data). Bit 6 set = IP-relative fixup (near CALL). Let `group_ofs(m) = m->final_code_ofs - (m->group_code_para << 4)` (byte offset of module's code start within its group's segment). The reference segment (`rref_seg`) is resolved:
  - 0 (intra-module code, non-relative): `fixup_val = group_ofs(m)` — adjusts the compiler's module-relative offset to be group-relative.
  - 0 (intra-module code, IP-relative): `fixup_val = m->final_code_ofs` — absolute in combined code buffer.
  - 1 (data): `fixup_val = final_data_ofs` of this module.
  - 2 (BSS): `fixup_val = total_data_len + final_bss_ofs` of this module.
  - ≥3 (import, code, non-relative): `fixup_val = sym->offset + group_ofs(sym->owner)` — group-relative offset of the target symbol.
  - ≥3 (import, code, IP-relative): `fixup_val = sym->offset + sym->owner->final_code_ofs`.
  - ≥3 (import, data): `fixup_val = sym->offset + sym->owner->final_data_ofs`.
  - For IP-relative: subtract `(patch_addr + width)` from fixup_val.

- **SEGRELOC** (type 6): patch the 2-byte segment word. `fixup_val` is the 0-based paragraph index within the load image (after header). The MZ loader adds the actual load paragraph at runtime.
  - Code reference → `m->group_code_para` (this module's .om group CS paragraph)
  - Import to code symbol → `sym->owner->group_code_para` (target group's CS paragraph)
  - Data/BSS reference or import → `total_code_len >> 4` (data segment paragraph)
  - MZ relocation entry location: for code-resident word, `seg = m->group_code_para`, `ofs = (m->final_code_ofs - group_start) + rofs` (offset within the group's segment).

**Entry point**: `find_symbol("start")` → `entry_ip = sym->offset + group_ofs(sym->owner)` (group-relative), `entry_cs_para = sym->owner->group_code_para`.

### 5. Write EXE (`olink_write_exe`)

- Compute `code_para`, `data_bss_para`, `hdr_para`.
- Write 28-byte MZ header.
- Write MZ relocation table (4 bytes per entry: `ofs`, `seg`).
- Pad to `hdr_para * 16`.
- Write code buffer, then data buffer.
- Print `NAME: Code=N, Data=N, BSS=N, Stack=N` to stdout.

## RDOFF2 relocation arithmetic

### Near CALL (E8 rel16) — IP-relative RELOC

```
Codegen stores:  field = -(patchAt + 2)          (addend, relative to field end)
Linker computes: field += symAddr - (patch_addr + width)
CPU executes:    IP = IP + field    (IP already points past instruction)
```

`patch_addr` = `final_code_ofs + rofs` (absolute offset in combined code buffer).  
`symAddr` = `sym->offset + sym->owner->final_code_ofs`.

### FAR CALL (9A oo oo ss ss) — SEGRELOC pair

Every FAR call to an imported symbol requires two RDOFF2 records:
- **RELOC** (non-relative): patches the 2-byte offset field (`oo oo`). `fixup_val = sym->offset + group_ofs(sym->owner)` — group-relative offset of the target (CS will point to the target group's start by the SEGRELOC).
- **SEGRELOC**: patches the 2-byte segment field (`ss ss`). `fixup_val = sym->owner->group_code_para` (0-based from load image start). Also emits an MZ relocation entry.

### SEGRELOC for data access

A SEGRELOC pointing at the data segment uses `fixup_val = total_code_len >> 4` — the paragraph index of the combined data segment within the load image.

## Data structures (olink.h)

### `TModule`
One per RDOFF2 object (including tar members). Tracks file name, byte offset within archive, segment sizes, and final layout offsets:

| Field | Type | Meaning |
|-------|------|---------|
| `group_id` | `int` | .om group identifier (same for all .rdf in one .om; unique per standalone .rdf) |
| `final_code_ofs` | `uint32_t` | byte offset in combined code buffer |
| `group_code_para` | `uint16_t` | CS paragraph of the .om group (= group start offset >> 4); shared by all modules in the group |
| `final_data_ofs` | `uint32_t` | byte offset in combined data buffer |
| `final_bss_ofs` | `uint32_t` | byte offset within combined BSS region |
| `used` | `int` | reachability flag (set by smart-link) |

### `TSymbol`
One per GLOBAL record. Linked list (`sym_head`). Name, owning module, `seg_id` (0=code, 1=data, 2=BSS), offset within that segment.

### `TMZReloc`
One per SEGRELOC record. Linked list (`reloc_head`/`reloc_tail`). `seg` and `ofs` written into the MZ relocation table (both 0-based from load image start, not from file start).

### `TImportNode`
Per-module, per-linking-pass temporary map from RDOFF2 import segment ID to resolved `TSymbol*`. Freed after each module's pass 2.

### `LinkerState`
All global linker state. Passed by pointer to every function. Initialised by `olink_init`, freed by `olink_free`.

## Implementation notes

- **C89, single compilation unit** (`olink.c` + `olink.h`). No dynamic dispatch, no global variables outside `main`'s `LinkerState`.
- **`patch_mem` width=4** is a no-op (32-bit fixups are out of scope for this 16-bit target).
- **`read_string`**: reads a NUL-terminated name from the current file position into a fixed buffer. The `ctx` parameter is accepted but unused (reserved for future error reporting).
- **`str_ends_with`**: case-insensitive suffix comparison. Used to recognise `.OM` extension as an alternative tar detection path (in case the ustar magic check fails on truncated files).
- **Module scan order**: modules are linked in the order they were scanned (depth-first tar traversal). The entry point `start` resolves to `entry_ip` within its own code segment; `start` does not need to be the first module.
- **No duplicate symbol detection**: if two modules export the same name, the first registered wins (`find_symbol` returns the first match in the singly-linked list). This is by design — it mirrors RDOFF2 convention.
- **malloc for code_buf**: `malloc(total_code_len)` where `total_code_len` may exceed 65000 bytes. On DOS with Open Watcom large model (`-ml`), `malloc` returns far pointers and can allocate more than 64KB. On the host (Linux/Mac), this is always fine. The data buffer (`total_data_len`) is separately guarded to ≤65520 bytes.
- **Stack size**: `STACK_SIZE = 8192` bytes (defined in `olink.h`). The stack segment is separate from data+BSS (large model). `SP` is initialised to `STACK_SIZE - 2`.

## Known portability issues (olink.c)

These are tracked in `docs/implementation-rules.md`:

| Issue | Location | Fix needed |
|-------|----------|------------|
| `malloc(total_code_len)` with `uint32_t` arg | `olink_perform_linking` | Guard: `if (ls->total_code_len > 65000u) olink_error(...)` on near-model builds only; large model is fine |

On Open Watcom large model (`-ml`), far `malloc` handles allocations larger than 64KB, so the guard is not strictly required for the DOS target. It is a hygiene issue for near-model or hosted builds.
